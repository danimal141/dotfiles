# Output Format Reference

## Markdown出力の品質基準

### 良いサマリの特徴

1. **自分の言葉で書かれている**: 原文のコピーではなく、概念を理解した上で再構成されている
2. **実践的**: 「明日から使える」レベルの具体性がある
3. **構造的**: 概念間の関係性が明確で、全体像の中での位置づけがわかる
4. **簡潔**: 冗長な説明を避け、エッセンスに絞っている

### 避けるべきパターン

- 原文の直接引用（著作権の問題）
- コード例のそのままの転載
- 「この章では〜について説明しています」のような内容の羅列だけのサマリ
- 抽象的すぎて何も行動に移せない記述

### 「実践への示唆」セクションの書き方

各章の「実践への示唆」は、以下のフォーマットで書くと効果的：

- **誰が**: どんな役割の人が
- **いつ**: どんな場面で
- **何を**: 具体的に何をすべきか

例（良い例）:
> コードレビュー時に、関数の責務が単一かどうかを最初のチェックポイントにする

例（悪い例）:
> 関数は小さく保つべきだということを意識する

### クイックリファレンステーブルの粒度

概念一覧テーブルは、以下を目安に項目数を調整する：

- 短い本（〜200ページ）: 10-15項目
- 中程度の本（200-400ページ）: 15-25項目
- 長い本（400ページ超）: 25-40項目

各項目の「定義（一言）」は15文字以内を目標にする。

## 言語の判定ルール

- PDFの本文が英語 → サマリは日本語で書く（ユーザーが日本語話者のため）
- PDFの本文が日本語 → サマリは日本語
- ユーザーが特定の言語を指定 → その言語に従う
- 技術用語は原語のまま残すことを推奨（例: "Single Responsibility Principle" は日本語訳と併記）
