# Tech & AI トレンドレポート生成スキル

ユーザーが指定した期間のAI・テック動向を、15〜20のキュレーション済みソースから
収集し、包括的な構造化Markdownレポートとして生成する。

## ステップ0：参照ファイルの読み込み

作業開始前に、必ず以下の2ファイルを読み込むこと：

1. `references/sources.md` — キュレーション済みソースリストと検索クエリパターン
2. `references/report-template.md` — 準拠すべきレポート構造

これらのファイルはこのSKILL.mdと同じディレクトリにある。

## ステップ1：時間範囲の解析

ユーザーのリクエストから具体的な日付範囲を抽出する。以下のパターンに対応：

| ユーザーの表現 | 解釈 |
|--------------|------|
| 「先週」 | 前週の月曜〜日曜 |
| 「今週」 | 今週の月曜〜今日 |
| 「3月」「2026年3月」 | 2026年3月1日〜31日 |
| 「2026-03-01〜2026-03-15」 | 指定された範囲そのまま |
| 「昨日」 | 前日1日分 |
| 「直近3日」 | 今日から3日前〜今日 |

曖昧な場合はユーザーに確認を求める。解釈した範囲は必ず確認すること：
「**{開始日}** 〜 **{終了日}** の期間でよろしいですか？」

レポートタイトル用の `period_label` を生成する。例：
- 「2026年3月3日〜3月9日」（週間）
- 「2026年3月」（月間）
- 「2026年3月15日」（1日分）

## ステップ2：検索クエリの設計

ソースリスト全体を幅広くカバーするため、15〜20件のweb_searchクエリを設計する。
個人ブログとarxiv論文に重点を置きつつ、全カテゴリを網羅することが目標。

### クエリ設計の原則

- クエリは短く具体的に：3〜6語＋日付修飾子
- 古い結果を避けるため、必ず年を含める
- 同種のソースはできるだけ1つの広めのクエリにまとめる
  （例：「AI model releases {月} {年}」で複数企業ブログの結果を取得）
- AI系個人ブログ（カテゴリ5a）はレポートの核。最低4クエリを割り当て、
  著者名を2〜3人ずつまとめて検索する
- arxiv・論文（カテゴリ2b）は分野別に最低4クエリを割り当て。
  LLM、エージェント、効率化、マルチモーダルなど軸を変えて検索する
- 2週間を超える期間の場合、サブ期間への分割を検討

### 推奨クエリ配分

個人ブログとarxiv論文を重点的に拾う配分。メディアは最小限に抑え、
現場の手触りが伝わる一次情報を優先する。

| カテゴリ | クエリ数 | 例 |
|---------|---------|-----|
| テックメディア（総合） | 1〜2 | `AI news highlights March 2026` |
| arxiv・論文（分野別） | 4〜6 | `arxiv LLM language model March 2026`, `arxiv AI agent reasoning March 2026` |
| 企業発表 | 2〜3 | `OpenAI Anthropic Google AI release March 2026` |
| ビジネス・市場 | 1 | `AI industry funding acquisition March 2026` |
| AI系個人ブログ（5a） | 4〜5 | `Simon Willison LLM March 2026`, `Chip Huyen blog March 2026`, `Hamel Husain fine-tuning March 2026` |
| システム系個人ブログ（5b） | 1〜2 | `Mitchell Hashimoto AI coding March 2026` |
| ニュースレター | 1 | `Andrew Ng Batch AI newsletter March 2026` |

## ステップ3：検索実行と情報収集

設計したweb_searchクエリを実行する。各クエリについて：

1. 返されたスニペットから、指定期間内の関連コンテンツを選別
2. スニペットが有望だが詳細が不足している場合、web_fetchで全文取得
   （ただし選択的に。web_fetchは合計5〜8回まで）
3. 結果が得られたソースと得られなかったソースを記録

情報収集と並行して、以下のレポートセクションに分類・整理する。
**各項目には必ずソースURLを併記すること**（レポート執筆時にリンク化するため）：
- モデルリリース・製品ローンチ（日付＋公式発表URL付き）
- 注目論文・研究（arxiv ID＋arxiv URL付き）
- 現場の声・実験メモ（個人ブログの記事URL付き）
- ソース別トピック（各記事URL付き）
- 新興トレンドパターン

個人ブログの記事を見つけた場合、web_fetchの優先度を上げて全文を取得する。
メディア記事よりブログ記事のweb_fetchを優先すること。

### 品質フィルター

- 指定期間内の情報のみを含める
- 二次報道より一次ソースを優先
- 複数ソースが同じイベントを報じている場合、最も信頼性の高いソースに統合
- 些末なアップデート（パッチリリース、軽微なUI変更など）は、ユーザーが
  詳細なカバレッジを明示的に求めない限りスキップ

## ステップ4：レポート執筆

`references/report-template.md` のテンプレートに厳密に従う。主なガイドライン：

### リンク原則

レポート内のすべての参照はクリッカブルなMarkdownリンクにすること。
Obsidianやその他のMarkdownビューアでそのまま飛べる状態にする。

- **出典**: プレーンテキストではなく `[記事タイトル](URL)` 形式にする
  - 良い例: `出典: [AI should help us produce better code](https://simonwillison.net/2026/Mar/10/...) (2026-03-10)`
  - 悪い例: `出典: simonwillison.net (2026-03-10)`
- **arxiv ID**: 必ずリンク付きにする — `[arXiv:2603.12229](https://arxiv.org/abs/2603.12229)`
- **モデルリリーステーブル**: リリース名を公式発表ページへのリンクにする
  - 例: `[Nemotron 3 Super 120B](https://developer.nvidia.com/blog/...)`
- **ソース別キートピック**: ソース名をサイトURLへのリンクに、トピック名を
  個別記事URLへのリンクにする
  - 例: `#### [Simon Willison](https://simonwillison.net/)`
  - 例: `- **[記事タイトル](URL)**: 要約`
- **トレンド分析内の参照**: 本文中のarxiv ID・ツール名・記事への言及もリンク化
- **GitHub リポジトリ**: `[リポジトリ名](https://github.com/owner/repo)` 形式

検索・web_fetch時にURLを必ず記録し、レポート執筆時に埋め込むこと。
URLが取得できなかったソースのみプレーンテキストで記載してよい。

### 言語

- レポートはユーザーが使用している言語で執筆（日本語での会話なら日本語、
  英語なら英語）
- 技術用語（モデル名、論文タイトル、ツール名）は言語に関係なく原語のまま

### エグゼクティブサマリー

- 最もインパクトのある動向から書き始める
- 3〜5文に収める
- 今後の注目ポイントで締める

### 主要モデルリリース＆製品ローンチ

- 一覧性を重視しテーブル形式で記載
- 時系列順
- 真に重要なリリースのみ — すべてのマイナーアップデートは不要

### 注目論文＆研究

- arxivを分野別に検索し、実務者コミュニティで反響のあった論文を選定
- ML専門外のシニアエンジニアにも理解できる平易な要約を提供
- 「従来Xだったのが、この手法でYになる」という変化の構造で書く
- 個人ブログでの言及があれば「実務者の反応」として記載
- 可能な限りarxiv IDと論文リンクを付記

### 現場の声・実験メモ

- レポートの差別化ポイント。メディア記事では得られない生々しいシグナルを集める
- 個人ブログから「Xを試してみたらYだった」「Zの使い勝手はこう変わった」
  のような一次体験を抽出
- 著者の評価・感想のエッセンスを忠実に伝える（ポジティブ/ネガティブ両方）
- 最低3件、理想は5件以上。このセクションが空の場合は検索を追加する

### ソース別キートピック

- テンプレートに示されたカテゴリごとにグループ化
- 該当する情報がなかったソースは省略（水増し不要）
- 各トピックは1〜2文の要約にとどめる
- このセクションは参照用インデックス — スキャンしやすさを重視

### 業界トレンド分析

- 最も価値のあるセクション — 最も労力をかける
- 複数ソースから浮かび上がる3〜5の横断的テーマを特定
- 各トレンドを収集データの具体例で裏付ける
- エンジニアや技術リーダーへの実践的な示唆を含める
- 漠然とした一般論は避け、何が変わったか・なぜ重要かを具体的に記述

### カバレッジノート

- シグナルが強かったソースとなかったソースを透明に報告
- 情報のギャップを正直に記載 — 読者が欠落箇所を把握できるようにする

## ステップ5：保存と配信

1. レポートを.mdファイルとして `/Users/hideaki.ishii/Documents/Obsidian/hideaki_ishii_main_vault/notes/tech-trend-reports/` に保存
   （ディレクトリが存在しない場合は作成する）
2. ファイル名は `{YYYYMMDD}_to_{YYYYMMDD}_report.md` 形式を使用
   （例：`20260303_to_20260309_report.md`）
3. ユーザーにファイルパスを提示
4. 提示後、ファイルを開かなくてもすぐに価値が得られるよう、
   主要ハイライトを2〜3文で口頭サマリーする

## 重要な注意事項

- **著作権**: 検索結果を参照する際は必ず言い換える。いかなるソースからも
  14語を超える逐語引用は行わない。
- **即時性**: レポートはweb_searchが返す結果の範囲内でのみ最新。直近1〜2日の
  期間の場合、カバレッジノートでこの制約に言及する。
- **スコープ制御**: 3ヶ月以上の長期間を求められた場合、品質向上のため月次
  レポートへの分割を提案する。順次実行を提案すること。
- **反復可能性**: レポート配信後、セクションの拡充、ソースの追加、
  分析の深度調整について提案する。
