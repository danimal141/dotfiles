# コードベース・ディープダイブ

大規模コードベースに対して、構造化されたマルチフェーズ調査を実行する。
目標は**エビデンスに基づく理解**を得ること。印象やフィーリングではない。

このスキルは**大規模な実世界のコードベース**（モノリス、モノレポ、マルチサービス構成）を
想定している。全ファイルを一度に読むことはできないため、
**段階的な絞り込み戦略**を採る：広域スキャン → 重点的な深掘り → 統合。

## 使うべきとき

- 初めて触るプロジェクトにオンボーディングするとき
- 大規模リファクタリングやマイグレーションの計画前
- システム全体の技術的負債を評価するとき
- 機能がシステムをどう端から端まで流れるか理解したいとき
- アーキテクチャドキュメントやADR（Architecture Decision Record）の作成時

## 使うべきでないとき

- 小さなスクリプトや単一ファイルのプロジェクト（直接読めばよい）
- すでに包括的なアーキテクチャ情報を含むCLAUDE.mdが存在する場合
- 特定のバグ修正だけが目的の場合（通常のデバッグワークフローを使う）

## ツールモード: 標準モード と Serenaモード

このスキルは2つのモードで動作する。フェーズ0で自動判定する。

### 標準モード（デフォルト）
`grep`, `find`, `Glob`, `Read` などClaude Code標準のツールのみで分析する。
Serena MCPが未接続の環境で使用する。

### Serenaモード（推奨・自動昇格）
Serena MCP が接続されている場合に自動的にこちらを使う。
LSP（Language Server Protocol）ベースのセマンティック分析により、
grep/findでは得られない**シンボルレベルの正確な依存関係**を把握できる。

**Serenaモードが有効な場合の主な利点**:
- `find_symbol`: クラスや関数の定義場所をLSPで正確に特定
- `find_referencing_symbols`: あるシンボルを参照している全箇所を構造的に取得
- `get_symbols_overview`: ファイル内のトップレベルシンボル一覧を取得
- `onboarding`: プロジェクト構造と重要タスクの自動認識
- `write_memory` / `read_memory`: 分析結果をSerenaのメモリに保存し次回以降に引き継ぎ
- `search_for_pattern`: プロジェクト全体でのパターン検索（grepの上位互換）

**Serenaが得意でない領域**（標準モードにフォールバック）:
- IaC（Terraform, HCL, docker-compose.yml）の分析
- CI/CDワークフロー（.github/workflows/*.yml）の分析
- Gitヒストリの考古学的分析
- ファイル行数カウントなどの単純な定量分析

## 分析フェーズ

フェーズは順番に実行する。各フェーズは前のフェーズで得たエビデンスの上に構築される。
発見事項はフェーズごとにレポートファイルに書き出し、コンテキストが溢れても部分的な成果が残るようにする。

### フェーズ 0: 準備とツール検出

分析を始める前に、出力ファイルを作成し、利用可能なツールを検出する。

```bash
mkdir -p .claude/reports
touch .claude/reports/codebase-analysis.md
```

#### Serena検出

以下の順で Serena MCP の利用可否を判定する：

1. **MCP接続の確認**: `mcp__serena__` プレフィックスのツールが利用可能か確認する。
   具体的には `mcp__serena__get_current_config` や
   `mcp__serena__check_onboarding_performed` を呼び出して応答があるか確認する。
2. **オンボーディング状態の確認**: Serenaが接続されている場合、
   `mcp__serena__check_onboarding_performed` でプロジェクトのオンボーディング済みか確認する。
3. **未オンボーディングの場合**: `mcp__serena__onboarding` を実行する。
   これによりSerenaがプロジェクト構造をインデックスし、
   `.serena/memories/` に初期分析結果を保存する。
   （初回は数分かかる場合がある。）

#### モード決定

| 条件 | モード | レポート記載 |
|------|--------|-------------|
| Serena MCPが接続＋応答あり | **Serenaモード** | `分析モード: Serena（LSPセマンティック分析）` |
| Serena MCPが未接続またはエラー | **標準モード** | `分析モード: 標準（grep/find）` |

レポートのヘッダーを初期化する：

```markdown
# コードベース分析レポート
生成日時: [日付]
リポジトリ: [名前]
分析者: Claude Code (codebase-deep-dive スキル)
分析モード: [Serena（LSPセマンティック分析） / 標準（grep/find）]
```

### フェーズ 1: 鳥瞰図（プロジェクトの正体）

**目標**: このプロジェクトが「何者か」を60秒で把握する。

#### 標準モード
1. ルートの `README.md`（または同等のもの）を読む
2. 依存関係マニフェストを読む（`package.json`, `Gemfile`, `go.mod`,
   `build.gradle`, `Cargo.toml`, `pyproject.toml` など、存在するもの）
3. ルートと第1階層のディレクトリを `ls` で確認する
4. `docker-compose.yml`, `Dockerfile`, `Makefile`, `Procfile` の有無を確認する
5. `.env.example` や `.env.sample` を確認し、必要な外部サービスを把握する
6. `CLAUDE.md` や `.claude/settings.json` があれば読む

#### Serenaモード追加手順
上記に加えて：
- `mcp__serena__onboarding` の結果がすでにある場合、
  `.serena/memories/` 内のオンボーディングメモリを `mcp__serena__read_memory` で読む。
  Serenaが自動認識した「プロジェクト構造」「ビルド/テスト方法」を活用する。
- `mcp__serena__list_dir` でルートのディレクトリ構造を確認する
  （標準のlsの代わりに使ってもよい）。

**抽出すべき情報**:
- プロジェクト名と目的（1文で）
- 主要言語とフレームワーク
- ランタイム依存（DB、キャッシュ、キュー、外部API）
- デプロイモデル（コンテナ、サーバーレス、従来型）
- モノレポか単一アプリか

**発見事項** をレポートの `## 1. プロジェクト概要` に記載する。

### フェーズ 2: アーキテクチャマップ（構造）

**目標**: コードの構成と、コンポーネント間の関係を把握する。

#### 標準モード
1. `find` や `Glob` でディレクトリ構造を2〜3階層の深さでマッピングする：
   ```bash
   find . -type d -maxdepth 3 \
     -not -path '*/node_modules/*' \
     -not -path '*/.git/*' \
     -not -path '*/vendor/*' \
     -not -path '*/dist/*' \
     -not -path '*/tmp/*' \
     | head -100
   ```
2. アーキテクチャパターンを特定する：
   - MVC: `app/models/`, `app/controllers/`, `app/views/`
   - レイヤードアーキテクチャ: `domain/`, `application/`, `infrastructure/`, `presentation/`
   - フィーチャーベース: `features/X/`, `modules/X/`
   - モノレポ: `packages/`, `apps/`, `services/`

   詳細は `references/architecture-patterns.md` を参照すること。

3. モノレポの場合、各パッケージ/サービスの役割を特定する
4. IaC（Infrastructure as Code）を探す（**常に標準モードで実行**）：
   ```bash
   find . -type f \( -name "*.tf" -o -name "*.hcl" \) | head -20
   find . -path "*/k8s/*" -o -path "*/kubernetes/*" -o -path "*/.github/workflows/*" | head -20
   ```
5. エントリーポイントを特定する

#### Serenaモード追加手順
上記に加えて、以下でコンポーネント間の**実際の依存関係**を解明する：

1. **主要ファイルのシンボル概観**: フェーズ1で特定した主要ディレクトリ内の
   代表ファイルに対して `mcp__serena__get_symbols_overview` を実行。
   各ファイルにどんなクラス・モジュール・関数が定義されているか一覧を得る。
2. **コンポーネント間依存の追跡**: 中核となるクラス（例: `ApplicationController`,
   `BaseService`, `DatabaseClient`）を `mcp__serena__find_symbol` で特定し、
   `mcp__serena__find_referencing_symbols` でそれを参照する全コンポーネントを
   リストアップする。これにより「実際に何が何に依存しているか」がgrepでは不可能な
   精度で判明する。
3. **レイヤー違反の検出**: 例えばプレゼンテーション層のファイルから
   インフラ層のシンボルへの直接参照がないかを
   `mcp__serena__find_referencing_symbols` で確認する。

**発見事項** をレポートの `## 2. アーキテクチャ` に記載する。
高レベルなコンポーネント関係のASCII図を含めること。
Serenaモードの場合、依存関係図はシンボルレベルのエビデンスに基づくものになる。

### フェーズ 3: 主要機能（ビジネスロジック）

**目標**: ユーザー/ビジネスの視点から、このシステムが「何をするか」を理解する。

#### 標準モード
1. 全ルート/エンドポイントをマッピングし、機能の表面積を把握する：
   ```bash
   # Rails
   grep -rn "get\|post\|put\|patch\|delete\|resources\|resource" config/routes.rb 2>/dev/null
   # Express/Koa
   grep -rn "router\.\(get\|post\|put\|delete\)" --include="*.ts" --include="*.js" | head -50
   # Next.js App Router
   find . -path "*/app/*/page.*" -o -path "*/app/*/route.*" | head -50
   # Go
   grep -rn "HandleFunc\|Handle\|GET\|POST" --include="*.go" | head -50
   ```
2. コアとなるドメインモデル/エンティティを特定する
3. 主要な機能領域ごとに、代表的なコントローラ/ハンドラを**1つ**読み、
   フロー（リクエスト → バリデーション → ビジネスロジック → レスポンス）を理解する
4. バックグラウンドジョブとスケジュールタスクを探す
5. 外部連携を特定する

#### Serenaモード追加手順
標準モードのルートマッピングに加えて、以下で機能フローを**端から端まで**追跡する：

1. **コントローラ→サービス→リポジトリの呼び出しチェーン追跡**:
   ルートマッピングで見つかった主要コントローラに対して
   `mcp__serena__get_symbols_overview` でアクション一覧を取得。
   代表的なアクション（例: `create`, `update`）の中で呼ばれているメソッドを
   `mcp__serena__find_symbol` で追跡し、サービス層→リポジトリ層/モデル層まで
   辿る。
2. **ドメインモデルの参照マップ**: 主要なモデルクラスに対して
   `mcp__serena__find_referencing_symbols` を実行し、
   「このモデルはどの機能から、どのように使われているか」を把握する。
   これにより「物件データ」「ユーザーデータ」などのドメインオブジェクトが
   システム内をどう流れているか可視化できる。
3. **イベント/コールバックの追跡**: `after_save`, `after_create` などの
   コールバックや、イベント発行元を `mcp__serena__find_symbol` で特定し、
   副作用のチェーンを追跡する。

**発見事項** をレポートの `## 3. 主要機能` に記載する。
ドメイン領域ごとにグルーピングし、各機能について目的・主要ファイル・データフローを記録する。
Serenaモードの場合、各機能の呼び出しチェーンを具体的なシンボル名で記載する。

### フェーズ 4: データ層（スキーマとモデル）

**目標**: データモデルを理解する。データモデルはあらゆるシステムの骨格である。

フレームワーク別の詳細な探索パターンは `references/schema-patterns.md` を参照すること。

#### 標準モード
1. スキーマ定義を見つける（schema.rb, schema.prisma, マイグレーション等）
2. コアテーブル/コレクションとそのリレーションを特定する
3. データアクセスパターンを調べる（N+1リスク、複雑なクエリ）
4. キャッシュ層を確認する
5. 検索インフラを確認する

#### Serenaモード追加手順
スキーマ定義の読み取り自体は標準モードで行う（DDLやスキーマファイルの読み取りは
テキストベースで十分）。その上で以下を追加する：

1. **モデルクラスの完全なインターフェース取得**:
   主要モデルに対して `mcp__serena__get_symbols_overview` を実行し、
   スコープ、バリデーション、コールバック、カスタムメソッドを含む
   **モデルの全体像**を取得する。
2. **リレーションの実使用パターン**: 主要モデルのアソシエーション
   （`has_many`, `belongs_to` 等）に対して
   `mcp__serena__find_referencing_symbols` を実行。
   「このリレーションは実際にどこで使われているか」を特定する。
   スキーマ上は存在するが実コードでは使われていないリレーションを検出できる。
3. **クエリパターンの深掘り**: N+1リスクが疑われるコードに対して、
   `mcp__serena__find_symbol` でクエリメソッドの定義を読み、
   `includes` / `preload` の有無を確認する。

**発見事項** をレポートの `## 4. データ層` に記載する。
最重要テーブル10〜15個の簡易ER図（ASCIIまたはMermaid）を含めること。

### フェーズ 5: 課題と技術的負債

**目標**: 具体的な問題点と懸念事項を特定する。

#### 標準モード（Serenaの有無に関わらず常に実行）
以下は定量分析であり、常に標準ツールで実行する：

1. 定量的シグナル：
   ```bash
   # 巨大ファイル（複雑性リスク）
   find . -name "*.rb" -o -name "*.ts" -o -name "*.js" -o -name "*.go" -o -name "*.py" \
     | grep -v node_modules | grep -v vendor \
     | xargs wc -l 2>/dev/null | sort -rn | head -20

   # TODO/FIXME/HACK の密度
   grep -rn "TODO\|FIXME\|HACK\|XXX\|WORKAROUND" \
     --include="*.rb" --include="*.ts" --include="*.js" --include="*.go" --include="*.py" \
     | grep -v node_modules | grep -v vendor | wc -l
   ```
2. テストカバレッジの評価
3. 依存関係の健全性（npm outdated, bundle outdated, npm audit 等）
4. コードスタイルの一貫性（リンター設定、TypeScript strictness、@ts-ignore数）
5. Git考古学：
   ```bash
   # 最も頻繁に変更されるファイル（churn = 複雑性リスク）
   git log --oneline --since="6 months ago" --name-only 2>/dev/null \
     | grep -E "\.(rb|ts|js|go|py)$" | sort | uniq -c | sort -rn | head -20
   # 直近のコントリビューター（バスファクター）
   git shortlog -sn --since="6 months ago" 2>/dev/null | head -10
   ```

#### Serenaモード追加手順
定量分析の結果を受けて、以下で**構造的な課題**を深掘りする：

1. **巨大ファイルの複雑度分析**: フェーズ5-1で特定した巨大ファイルに対して
   `mcp__serena__get_symbols_overview` を実行。シンボル数が多すぎるファイルは
   「神クラス」の候補。単一クラスに何メソッドあるかを定量化する。
2. **高churnファイルの依存分析**: Git考古学で特定した高churnファイル内の
   主要シンボルに対して `mcp__serena__find_referencing_symbols` を実行。
   「このファイルが頻繁に変わるのは、多くのコンポーネントが依存しているからか？」
   を検証する。参照元が多い＝変更の影響範囲が広い＝リスクが高い。
3. **循環参照の検出**: フェーズ2で特定したコンポーネント間で、
   A→BかつB→Aの参照関係がないかを `mcp__serena__find_referencing_symbols` で
   確認する。
4. **デッドコードの検出**: 主要モジュールのpublicメソッドに対して
   `mcp__serena__find_referencing_symbols` を実行し、参照が0のシンボルを
   リストアップする。

**発見事項** をレポートの `## 5. 課題と技術的負債` に記載する。
課題は深刻度で分類する：**致命的** / **高** / **中** / **低**。

### フェーズ 6: 総合分析と提言

**目標**: すべてをまとめ、アクション可能な理解に昇華する。

レポートの `## 6. 総合分析` に以下を記載する：
1. **システム要約** — これは何で、どう動くか（3〜5文）
2. **強み** — うまくやれている点。維持すべき良いパターン
3. **主要リスク** — 問題を起こしうるトップ3〜5の事項
4. **推奨アクション** — 発見事項に基づく優先順位付きの行動提案
5. **チームへの確認事項** — コードだけでは答えが出ない問い
   （ビジネスコンテキスト、歴史的な設計判断、今後の計画など）

#### Serenaモード追加手順
- 分析結果を `mcp__serena__write_memory` で保存する。
  メモリ名: `codebase-analysis-[日付]`
  内容: レポートの「6. 総合分析」セクションの要約。
  これにより次回のClaude Codeセッションで
  `mcp__serena__read_memory` から前回の分析結果を引き出し、
  差分分析や経時変化の追跡が可能になる。

## 実行ガイドライン

- **段階的に書く**: 各フェーズの完了後にレポートへ書き出す。コンテキストウィンドウが
  溢れても、ディスク上に部分的な成果が残る。
- **選択的に読む**: 1000ファイルのコードベースで全てを読むことはできない。
  代表的なファイルを読む。Grep と Glob でパターンを見つけ、最も情報量の多い
  ファイルだけを Read する。Serenaモードでは `get_symbols_overview` が
  ファイル全体を読まずにシンボル一覧を取得できるため、より効率的に選別できる。
- **不明点に正直に**: フェーズの結果が不明瞭な場合はそう書く。
  「認証フローが不明確 — セッション管理のコードが見つからなかった」は
  推測するよりも有用である。
- **スタックに合わせる**: 上記のシェルコマンドは例示である。実際の言語/フレームワークに
  合わせて調整する。GoプロジェクトにはRails固有のコマンドは不要。
- **Serenaの使い分け**: Serenaは**ソースコードのセマンティック分析**に強い。
  一方、設定ファイル（YAML, HCL, Dockerfile）、Git履歴、ファイル統計などは
  標準ツールの方が適している。「コードの意味を理解する」必要があるときはSerena、
  「ファイルの存在やパターンを探す」ときは標準ツールと使い分ける。
- **Serenaのトークン効率**: Serenaモードでは `find_referencing_symbols` 等が
  必要なシンボル情報だけを返すため、ファイル全体を読むよりトークン効率が良い。
  巨大ファイルの分析では積極的にSerenaツールを使うこと。
- **時間配分**: 全フェーズを10〜15分のアクティブワーク内で完了させることを目指す。
  フェーズ0（Serena検出含む）は2分以内。フェーズ1は2分以内。
  フェーズ2〜4がコア（各3〜5分）。フェーズ5は深掘り可能だが上限を設ける。
  Serenaモードはオンボーディングに初回数分かかるが、
  その後のフェーズではトークン効率が上がるため総時間は同程度になる。
